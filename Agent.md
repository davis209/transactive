# 系统简介

本系统是一个基于 CORBA 的大型分布式工业软件。应用于地铁监控，功能模块包括:
* PA(广播) 
* PIDS(乘客信息系统)
* Radio
* Train
* PMS
* ECS
* Alarm
* Event
* Duty/Rights

每个功能模块软件采用Agent + Manager架构，Agent为服务端，运行于Linux服务器， Manger为客户端，运行于Windows系统，它们之间通过CORBA协议通讯。
例如：
* PAAgent -> PA模块的服务程序
* PAManager -> PA模块的客户端程序
* PIDSAgent -> PIDS模块的服务程序
* PIDSManger -> PIDS模块的客户端程序

# 基本原则

开发过程中必须遵循以下原则：

## 1. 优先复用已有代码

不要重新实现已有功能。

修改之前：

先搜索：

* 是否已有类似类
* 是否已有类似接口
* 是否已有工具函数

如果已有，请优先复用。

---

## 2. 不要修改系统架构

不要：

* 更换通信协议
* 引入新的 RPC
* 修改总体架构

所有远程通信必须继续使用 CORBA。

---

## 3. 如非必要，不要引入新的第三方库

如非必要不要引入第三方库,请使用以下项目已有基础库:
* Boost
* ACE
* Curl
* omniORB

---

## 4. 保持代码风格一致

变量命名

成员变量：

m_xxx

静态变量：

s_xxx

类：

CAlarmManager

函数：

CamelCase

日志：

调用已有日志记录接口函数

不要：

printf()

cout

---

## 5. 修改范围尽量小

优先：

修改已有类

不要：

重新写一个新版本

除非明确要求。

---

# CORBA 开发原则

新增接口必须遵循：

1. 修改 IDL

2. 编译已有相关工程重新生成 Stub

3. 修改 Agent Impl

4. 修改 Manager Proxy

不要跳过任何步骤。

---

# 数据库读写原则

本项目已经对数据库访问接口进行了封装，相关类的名称形如 XXXAccessFactory, 代码基本上位于transactive/core/data_access_interface目录下，
你必须调用这些类中相关函数读写数据库，如果这些类缺少相关接口函数，你可以对这些类进行修改。

不要通过其它方式读写数据库。

---

# 修改代码之前

请先分析：

1. 哪几个模块受影响

2. 哪几个 Agent 受影响

3. 哪几个 Manger 受影响

4. 是否涉及数据库

5. 是否涉及 IDL

分析完成后再开始修改。

---
